# DB_SCHEMA.md

## 목적

이 문서는 BuildGraph 3주 MVP의 PostgreSQL 공통 DB 스키마 계약을 정의한다.
실제 실행 DDL은 `apps/api/src/main/resources/db/migration`의 Flyway migration 파일이며, 이 문서는 migration과 API/route 문서가 따라야 하는 공통 계약이다.

## 설계 원칙

- 내부 PK는 모든 테이블에서 `BIGINT GENERATED BY DEFAULT AS IDENTITY`를 사용한다.
- 외부 API path와 frontend route에 노출되는 리소스는 `public_id UUID NOT NULL UNIQUE DEFAULT gen_random_uuid()`를 사용한다.
- API path의 `{id}`와 API DTO의 `id`는 내부 PK가 아니라 `public_id`를 의미한다.
- 핵심 관계와 FK는 JSONB에 숨기지 않고 정규화한다.
- JSONB는 가변 속성, Tool 요청/응답 원문, RAG metadata, 후보 배열에만 제한적으로 사용한다.
- `products` 도메인은 사용하지 않고 `parts` 도메인을 사용한다.
- 주문, 결제, 배송, 재고 차감, 타임세일은 V1 범위에서 제외한다.
- LLM/RAG cache와 quota는 DB 테이블이 아니라 Redis/runtime 정책으로 관리한다.
- Google OAuth one-time code도 DB에 저장하지 않고 Redis/runtime에서 짧은 TTL로 관리한다.
- AWS 공용 DB는 검증 환경이며, 기준은 Git에 남는 Flyway/SQL 파일이다.

## 공통 컬럼 규칙

| 규칙 | 내용 |
|---|---|
| 내부 PK | 모든 테이블은 `id BIGINT` 내부 PK를 가진다. |
| 외부 ID | path로 조회되는 리소스는 `public_id UUID NOT NULL UNIQUE DEFAULT gen_random_uuid()`를 가진다. |
| UUID 생성 | Sprint 1은 DB 기본값 `gen_random_uuid()`를 기준으로 한다. 애플리케이션에서 직접 생성하더라도 DB default는 유지한다. |
| 시간 | 업무 테이블은 `created_at`, 필요 시 `updated_at`을 가진다. |
| soft delete | 업무 테이블 일부만 `deleted_at`을 가진다. 이력/근거 테이블에는 기본 적용하지 않는다. |
| 사용자 FK | V1은 전체 로그인 정책이므로 사용자 생성 데이터는 기본적으로 `user_id NOT NULL`이다. |

`public_id` 적용 테이블:

- `users`
- `requirements`
- `builds`
- `parts`
- `price_jobs`
- `agent_sessions`
- `tool_invocations`
- `rag_evidence`
- `agent_log_uploads`
- `as_tickets`

`deleted_at` 적용 테이블:

- `users`
- `parts`
- `price_alerts`
- `price_jobs`
- `compatibility_rules`
- `benchmark_summaries`
- `as_tickets`

## enum/status 규칙

DB와 API의 enum/status 값은 대문자 `SNAKE_CASE`를 사용한다. UI 문구는 별도 label로 분리한다.

| enum | 값 |
|---|---|
| User role | `USER`, `ADMIN` |
| Generic status | `ACTIVE`, `INACTIVE`, `DISCONTINUED` |
| Part category | `CPU`, `GPU`, `RAM`, `MOTHERBOARD`, `STORAGE`, `PSU`, `CASE`, `COOLER` |
| Tool status | `PASS`, `WARN`, `FAIL` |
| Confidence | `LOW`, `MEDIUM`, `HIGH` |
| Agent session status | `QUEUED`, `RUNNING`, `RAG_SEARCHED`, `TOOLS_CALLED`, `SUMMARY_READY`, `FALLBACK_READY`, `SUCCEEDED`, `FAILED`, `CANCELLED` |
| AS ticket status | `OPEN`, `ASSIGNED`, `IN_PROGRESS`, `RESOLVED`, `CLOSED`, `CANCELLED` |
| Log upload status | `UPLOADED`, `PROCESSING`, `FAILED` |
| Price alert status | `ACTIVE`, `TRIGGERED`, `DISABLED` |
| Price job status | `QUEUED`, `RUNNING`, `SUCCEEDED`, `FAILED`, `CANCELLED` |

`SUMMARY_READY`는 요약 생성 완료 중간 상태이고, `SUCCEEDED`는 Agent 전체 완료 최종 상태다.

## 상태 전이 규칙

MVP 기준 결정값:

- 아래 표에 없는 상태 전이는 허용하지 않고 API는 `409 CONFLICT_STATE`를 반환한다.
- 생성 전 상태는 `-`로 적는다.
- 실패 시 API status code가 `-`인 행은 허용 전이라 실패 응답이 없다.
- `audit log 기록 여부`는 `admin_audit_logs` 기록 기준이다. Agent 내부 실행 이력은 별도로 `agent_sessions.state_timeline`에 기록한다.
- public API나 audit metadata에는 내부 `BIGINT id`를 기록하지 않고 `public_id` 또는 stable key만 기록한다.

### build_items

목적: build에 포함된 부품 목록을 저장한다.

Owner: 1번

| 컬럼명 | 타입 | nullable | FK | 설명 |
|---|---|---:|---|---|
| `id` | `BIGINT` | no | - | 내부 PK |
| `build_id` | `BIGINT` | no | `builds.id` | build |
| `part_id` | `BIGINT` | no | `parts.id` | 부품 |
| `category` | `VARCHAR(50)` | no | - | CPU/GPU/RAM 등 |
| `price` | `INTEGER` | no | - | build 생성 당시 가격 |
| `created_at` | `TIMESTAMPTZ` | no | - | 생성 시각 |

Index:

- index: `build_items.build_id`
- index: `build_items.part_id`
- unique: `(build_id, category)`

MVP 기준 결정값:

- 한 build 안에는 category별 부품을 1개만 저장한다.
- 같은 `build_id`와 `category`에 여러 `build_items` row를 append하지 않는다.
- `POST /api/builds/{id}/change-part`는 같은 transaction 안에서 기존 category row를 물리 삭제한 뒤 새 `build_items` row를 insert한다. 내부 `build_items.id`는 public API에 노출하지 않으므로 교체 후 바뀌어도 된다.
- 교체 이력 테이블은 V1에서 만들지 않는다. 교체 전후 diff는 API 응답으로만 반환한다.
- category 값은 `parts.category`와 동일한 대문자 식별자를 사용한다. 예: `CPU`, `GPU`, `RAM`, `MOTHERBOARD`, `STORAGE`, `PSU`, `CASE`, `COOLER`.

|---|---|
| 최대 파일 크기 | 10 MiB. 초과 시 API는 `400 FILE_VALIDATION_ERROR`를 반환하고 row를 만들지 않는다. |
| 허용 MIME | `application/json`, `application/x-ndjson`, `text/plain`, `application/octet-stream` 중 확장자가 허용된 경우만 허용한다. |
| 허용 확장자 | `.jsonl`, `.ndjson`만 허용한다. `.json`, `.log`, 압축 파일은 V1에서 제외한다. |
| JSONL 검증 | 빈 파일 금지, 각 line은 UTF-8 JSON object여야 한다. line array, primitive, trailing comma, 파싱 실패 line이 있으면 전체 업로드 실패다. |

| 보관 기간 | `delete_after = created_at + 30 days`다. |
| 삭제 배치 책임 | 4번 owner가 삭제 대상 선정 query를 관리하고, 5번 owner가 scheduler/infra 실행을 승인한다. |
| 관리자 접근 기록 | 관리자가 업로드 상세 또는 연결 AS ticket 상세에서 로그 요약/원문 위치를 조회하면 `admin_audit_logs`에 `AGENT_LOG_VIEWED` action을 기록한다. |
| 원문 노출 범위 | public API는 원문 로그를 반환하지 않고 `summary`, `status`, `fileName`, `createdAt`, `deleteAfter`만 반환한다. admin API도 V1에서는 원문 다운로드 API를 만들지 않는다. |

## JSONB 최소 shape

아래 shape는 MVP 기준 결정값이다. 필드는 추가할 수 있지만 기존 필드명과 타입은 API DTO와 동시에 변경해야 한다.

### `requirements.parsed_context`

```json
{
  "usageTags": ["GAMING"],
  "budget": 1500000,
  "preferredVendors": ["NVIDIA"],
  "mustHave": ["WIFI"],
  "questions": [
    {
      "key": "monitorResolution",
      "answer": "QHD"
    }
  ]
}
```

| field | type | nullable | 설명 |
|---|---|---:|---|
| `usageTags` | `string[]` | no | 사용 목적. 예: `GAMING`, `OFFICE`, `AI_DEV` |
| `budget` | `number` | yes | 사용자가 제시한 예산 |
| `preferredVendors` | `string[]` | yes | 선호 제조사 또는 칩셋 |
| `mustHave` | `string[]` | yes | 필수 요구사항 |
| `questions` | `object[]` | yes | 추가 질문과 답변 |

## 보관/삭제 정책

- `users`, `parts`, `price_alerts`, `price_jobs`, `compatibility_rules`, `benchmark_summaries`, `as_tickets`는 `deleted_at`으로 soft delete한다.
- `agent_log_uploads`는 `delete_after` 기준으로 보관 만료를 관리한다. V1 기본 정책은 업로드 후 30일 보관이다.
- `refresh_tokens`는 삭제하지 않고 `revoked_at`으로 폐기한다.
- `rag_evidence`는 전체 원문 문서를 저장하지 않고 사용 chunk와 embedding만 저장한다.

## 병렬 개발 동결 및 변경 절차

MVP 기준 결정값:

- table name, column name, enum/status, `public_id` 적용 대상은 기능 구현 시작 전 동결한다.
- DB 변경이 API DTO, route ownership, 권한 정책에 영향을 주면 `API_CONTRACT.md`와 `ROUTE_OWNERSHIP.md`를 같은 PR에서 동시에 수정한다.
- API path의 `{id}`와 DTO의 `id`는 항상 `public_id`이며, 내부 `BIGINT id`는 DB 내부 FK와 join에만 사용한다.
- 새 table이 필요해 보이는 경우에도 V1 제외 범위에 속하면 추가하지 않는다.
- 상태 전이 추가/삭제는 이 문서의 상태 전이표와 `API_CONTRACT.md`의 409 조건을 함께 수정한다.
- JSONB shape 변경은 이 문서의 JSONB 최소 shape와 `API_CONTRACT.md`의 Schema Appendix를 함께 수정한다.

계약 테스트 최소 세트:

| 테스트 | DB 기준 |
|---|---|
| public API BIGINT 비노출 | API response와 audit `target_id`에 `BIGINT id`를 매핑하지 않는다. |
| 401/403 권한 분기 | 권한 자체는 API/security 기준이며, admin mutation은 `admin_audit_logs`에 남긴다. |
| 금지된 상태 전이 409 | 상태 전이표에 없는 변경은 저장하지 않는다. |
| soft delete 조회 제외 | `deleted_at IS NULL` 조건을 목록/상세 기본 조회 조건으로 둔다. |
| Flyway migration 순서 검증 | extension, parent, child, FK, seed 순서가 권장 migration 순서와 맞아야 한다. |
| JSONL 업로드 validation | 검증 실패 시 `agent_log_uploads` row와 파일을 남기지 않는다. |
| pagination 기본값/최대값 | DB query는 API의 `page=0`, `size=20`, `max size=100` 기준을 따른다. |

## Flyway migration 규칙

1. Flyway/SQL 파일이 DB 기준이다.
2. AWS 공용 DB는 migration 검증 환경으로만 사용한다.
3. AWS DB에서 직접 DDL을 수정하지 않는다.
4. 5번 담당자가 migration 번호, 실행 순서, 공통 enum, extension 순서를 승인한다.
5. 각 담당자는 자기 도메인 SQL 초안을 제출할 수 있지만 최종 번호 배정은 5번이 한다.
6. `pgvector` extension은 `rag_evidence.embedding`보다 먼저 생성되어야 한다.
7. `pgcrypto` extension은 `public_id DEFAULT gen_random_uuid()`보다 먼저 생성되어야 한다.
8. parent table은 child table보다 먼저 생성한다.
9. `agent_sessions.as_ticket_id` FK는 `as_tickets` 생성 이후 추가한다.
10. seed 기준 데이터는 Flyway migration으로 버전 관리한다.
11. 실제 Flyway DDL은 `apps/api/src/main/resources/db/migration`에 둔다.

권장 migration 순서:

```text
V1__extensions.sql
V2__users_auth.sql
V3__parts_price.sql
V4__quote_build.sql
V5__support_ticket.sql
V6__agent_rag_tool.sql
V7__admin_audit_seed.sql
```

현재 저장소에는 위 순서의 Flyway migration이 반영되어 있다. 기존 PostgreSQL volume이 남아 있으면 새 migration과 seed가 다시 실행되지 않으므로, 공통 DB를 처음부터 검증할 때는 `docker compose down -v` 후 `docker compose up --build`를 사용한다.
