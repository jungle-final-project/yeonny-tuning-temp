# PC 조합 종합 1000점 평가 정책

이 문서는 완성 PC 조합을 사용자에게 `0~1000점`으로 보여주기 위한 종합 평가 기준을 정의한다.

## 목적

- 기존 `benchmark_summaries.score`는 `0~100` 카테고리 내부 비교용 점수다.
- 새 `compositeScore.score`는 CPU, GPU만 평균내는 점수가 아니라 완성 견적 자체의 품질을 보여주는 사용자 표시용 점수다.
- 사용자 요청 예산을 얼마나 잘 지켰는지는 `compositeScore.score`에 섞지 않고 `requestFit`으로 별도 표시한다.
- 1000점 기준을 쓰는 이유는 시인성이다. 실제 계산은 5개 영역별 점수를 합산하고, 화면은 곡선/막대 그래프에서 `0~1000` 위치로 보여준다.
- 정확 FPS, 렌더링 시간, 고장 확률, 체감 만족도를 보장하지 않는다.

## 적용 위치

- API: `POST /api/build-graphs/resolve`
- 응답 필드: `compositeScore`
- 구현: `BuildCompositeScoreService`
- 저장: DB에 저장하지 않고 요청 시 현재 부품 DB, Tool 결과, 벤치/스펙 기준으로 계산한다.
- AI 설명: `POST /api/ai/build-chat`의 `EXPLAIN_BUILD_SCORE` 경로는 같은 `BuildEvaluationService` 결과를 `buildAssessment`로 반환한다. 클라이언트가 보낸 점수나 경고 문구는 근거로 사용하지 않으며, 문장 다듬기는 RAG 없는 소형 `gpt-5.4-mini` structured-output 호출로 제한한다.

## 1000점 구성

| 영역 | 배점 | 의미 |
|---|---:|---|
| 성능 | 430 | CPU/GPU benchmark, RAM, STORAGE, COOLER 스펙 기반 작업 성능 여유. 최상위 CPU/GPU 차이는 비선형 곡선으로 강조 |
| 호환·장착 안전성 | 220 | compatibility/power/size FAIL이면 전체 0점. 통과 시 만점, 장착 여유가 좁을 때만 일부 감점 |
| 병목·여유 균형 | 160 | CPU/GPU 급 차이, PSU headroom, 케이스 clearance |
| 확장·운영 여유 | 110 | PSU 규격/용량, 보드 확장성, 케이스 여유, 저장장치 구성 |
| 근거 신뢰도 | 80 | benchmark/FPS seed, Tool 판정, 카테고리별 스펙 보유율 |
| 합계 | 1000 | 사용자 표시용 종합 점수 |

## 차단/Cap 정책

합산 점수가 높아도 치명 조건이 있으면 최종 점수는 차단 또는 cap을 적용한다.

| 조건 | 최대 점수 | 이유 |
|---|---:|---|
| compatibility FAIL | 0 | 소켓, 메모리, 쿨러 호환 실패는 완성 PC가 아니다. |
| power FAIL | 0 | 파워 부족 조합은 완성 PC가 아니다. |
| size FAIL | 0 | 케이스 장착 불가 조합은 완성 PC가 아니다. |
| 장착 가능하지만 케이스 여유가 매우 좁음 | 880~960 | GPU 길이, CPU 쿨러 높이, PSU 깊이가 한계에 가까우면 조립/운영 여유가 낮다. |
| 끝판왕 조합 + 명시적으로 airflow 약한 케이스 | 929 | 고발열 조합에서 airflow 근거가 낮으면 장착 가능해도 최고 점수를 제한한다. |
| 핵심 카테고리 누락 1개 | 760 | 완성 견적이 아니므로 점수를 제한한다. |
| 핵심 카테고리 누락 2개 | 680 | 완성도가 크게 낮다. |
| 핵심 카테고리 누락 3개 이상 | 600 | 부품 후보 묶음으로만 본다. |

## 호환·장착 안전성 세부 원칙

- `compatibility`, `power`, `size` 중 하나라도 `FAIL`이면 최종 `compositeScore.score=0`이다.
- `FAIL`이 없으면 기본적으로 호환·장착 안전성은 만점이다.
- 다만 GPU 길이, CPU 쿨러 높이, PSU 깊이처럼 장착 자체는 가능하지만 여유가 좁은 `size WARN`은 장착 여유에 따라 일부 감점한다.
- `size WARN`의 장착 여유 기준은 GPU `20mm 미만`, 공랭 CPU 쿨러 `5mm 미만`, PSU `10mm 미만`이다. Tool 판정과 배치도 관계선은 이 기준을 함께 사용하며, `높이 157mm` 같은 부품 스펙 자체를 경고 사유로 표시하지 않는다.
- 케이스는 성능 벤치마크 부품이 아니다. 장착 불가는 0점, 장착 가능 후에는 GPU 길이, CPU 쿨러 높이, PSU 길이 여유 mm가 핵심이다.
- airflow는 `frontMesh`, `airflowFocus`, `airflow=HIGH`처럼 내부 자산에 명시된 경우에만 보조 cap 근거로 사용한다. 제품명에 `AIR`, `MESH`, `FLOW`가 들어간다는 이유만으로 airflow를 추정하지 않는다. airflow 근거가 없으면 임의 감점하지 않고 근거 신뢰도에서만 해석한다.
- airflow 필드가 아예 없으면 확장·운영 여유 점수에서도 임의 중립값을 넣지 않는다. 확인 가능한 GPU·CPU 쿨러 장착 여유에 기존 상대 가중치를 재분배한다.
- 케이스 cap 설명은 `케이스 여유 또는 airflow` 같은 혼합 문구를 쓰지 않는다. GPU 길이, CPU 쿨러 높이, PSU 깊이 중 실제 제한 원인을 선택해 `실제 치수 / 허용 치수 / 남은 mm`를 표시하고, airflow는 명시값이 `false`인 고발열 조합에서만 별도 `LOW_CASE_AIRFLOW` 사유로 설명한다.
- 파워는 `FAIL`이면 0점이다. `PASS`라도 정격에 딱 맞거나 부하율이 높은 구성은 완성 PC로는 가능하지만 끝판왕 점수 상한을 제한한다. 반대로 1000W 이상처럼 이미 충분한 여유가 있는 구간끼리는 점수 차이를 작게 둔다.
- 저장장치는 PCIe 세대와 순차 읽기/쓰기 성능을 성능 평가의 중심으로 둔다. 용량은 성능 점수에서는 낮은 보조 가중치만 갖고, 주로 운영 여유와 최고 점수 cap에서 반영한다. 2TB 미만, 1TB 미만, PCIe 3.0급 저장장치가 고성능 조합에 들어가면 단계별 최고 점수 상한을 제한한다. 단, 저장장치 부족은 호환 실패가 아니므로 전체 0점으로 처리하지 않는다.
- RAM은 총 용량 티어를 우선한다. 4GB, 8GB, 16GB, 32GB, 64GB, 128GB를 기본 체급으로 나누고, 같은 용량 안에서 듀얼채널, DDR5 여부, 클럭을 보조 가중치로 반영한다. 클럭이 없는 자산은 저성능으로 단정하지 않고 DDR 세대 기준 중립값으로 처리한다. 16GB 이하 또는 32GB 단일 구성은 고성능 조합의 최고 점수 상한을 제한한다.
- 쿨러는 CPU/GPU처럼 절대 성능 벤치마크로 보지 않는다. 현재 선택된 CPU의 TDP 대비 쿨러 TDP class, 공랭/수랭 체급, 라디에이터 크기, 히트파이프 근거를 통해 냉각 여유를 계산한다. 가격은 9만원 미만, 9~20만원, 20~40만원, 40만원 이상 체급 추정 보조값으로만 약하게 반영한다. CPU 발열 대비 여유가 낮으면 장착 가능하더라도 최고 점수 상한을 제한한다.
- 메인보드는 FPS를 직접 올리는 성능 부품으로 보지 않는다. socket, memoryType이 맞지 않으면 Tool 호환 실패로 전체 0점이며, 호환 통과 후에는 CPU/GPU 체급을 받쳐줄 수 있는 플랫폼인지로 판단한다. 9700X/5070 Ti 같은 중고급 조합에서는 B650/B760급도 큰 감점 없이 허용하지만, 9950X3D/RTX 5090 같은 끝판왕 조합에서는 X870E/Z890급이 아니면 최고 점수 상한을 둔다.
- B850/B860 또는 PCIe 5.0 + Wi-Fi를 갖춘 최신 B급 보드는 일반 B650/B760보다 높은 플랫폼 체급으로 보되, X/Z급과 같은 만점 플랫폼으로 보지는 않는다. 일반 B650/B760은 끝판왕 CPU/GPU 조합에서 S 라벨 직전으로 제한될 수 있고, A620/H610 같은 엔트리 보드는 고성능 CPU/GPU 조합에서 강한 상한을 적용한다.
- 근거 데이터가 부족한 경우는 이 영역에서 임의 감점하지 않고 `근거 신뢰도`에서 반영한다.

## 요청 예산 적합도 분리

- `compositeScore.score`는 견적 비교용 성능/완성도 점수이므로 가격 효율을 섞지 않는다.
- `800만원 요청에 1200만원 구성` 같은 상황에서도 견적 자체가 좋으면 높은 점수를 받을 수 있다.
- 대신 같은 응답의 `requestFit`이 `OVER_BUDGET`과 예산 차액을 표시한다.
- 단일 추천 생성 단계에서는 `requestFit`이 낮은 조합을 후보에서 제외하거나 경고로 표시할 수 있다.
- 여러 가격대 견적을 비교하는 화면에서는 `compositeScore.score`와 총액, `requestFit`을 분리해서 보여준다.
- 가성비가 필요한 화면은 `compositeScore.score`를 변형하지 말고 별도 `valueScore` 또는 정렬 옵션으로 계산한다.

## 데이터 사용 원칙

- `benchmark_summaries.score`는 CPU/GPU 등 카테고리 내부 성능 근거로만 사용한다.
- 성능 영역은 `benchmark_summaries.score`를 선형 합산하지 않는다. GPU/CPU 최상위 체급 차이가 화면에서 작게 보이지 않도록 성능 비교용 곡선을 적용한다.
- GPU는 가장 강하게 곡선을 적용한다. 예를 들어 원점수 100점급은 그대로 최상위로 두고, 95점급은 종합 성능 영역에서 더 낮게 환산되어 5090/5080 같은 상위 라인 차이가 시각적으로 드러난다.
- CPU도 비선형 곡선을 적용한다. 100점급 CPU와 90점대 CPU가 화면에서 비슷해 보이지 않도록 성능 영역 내 CPU 배점과 곡선을 강화한다.
- CPU는 AMD X3D 계열을 별도 처리한다. X3D는 게임/FPS 점수에서는 보너스를 받지만, 영상 편집, 렌더링, 컴파일 같은 작업 성능에서는 코어/스레드 기반 생산성 점수와 블렌딩한다. 따라서 7800X3D 같은 게임 특화 CPU가 작업용 종합점수를 과하게 가져가지 않고, 9950X3D처럼 코어 수도 충분한 상위 X3D만 끝판왕 CPU로 취급한다.
- SSD는 PCIe 5.0/고속 SSD와 PCIe 4.0, PCIe 3.0 이하가 화면에서 구분되도록 PCIe 세대와 속도에 높은 가중치를 둔다. 용량은 성능 자체보다 운영 여유/완성도 요소로 낮은 가중치와 cap에 반영한다.
- RAM은 용량이 성능/사용성의 1차 기준이다. 같은 32GB 안에서는 6000MHz/6400MHz 같은 클럭 차이를 작게 보고, 16GB와 32GB처럼 용량 체급이 다른 경우를 더 크게 반영한다. 클럭 데이터가 없는 수집 자산은 감점 근거로 쓰지 않는다.
- 쿨러는 360mm AIO, 240/280mm AIO, 상급 듀얼타워 공랭, 일반 공랭을 구분하되, 가장 중요한 기준은 현재 CPU와의 냉각 여유 비율이다. 가격 구간은 스펙 누락이나 동급 비교에서만 보조로 쓰며, 40만원 이상 제품은 LCD/RGB 프리미엄일 수 있어 20~40만원 고급형보다 냉각 성능을 자동으로 높게 보지 않는다. 기본 공랭이 고발열 CPU와 조합되면 최고 점수를 제한한다.
- 메인보드는 chipset tier, PCIe 세대, Wi-Fi/확장성 근거를 사용한다. X/Z급은 끝판왕 플랫폼으로, 최신 B급은 상급 mainstream 플랫폼으로, 일반 B급은 실사용 가능하지만 최상위 조합에서는 제한되는 플랫폼으로 해석한다. 메인보드 점수는 단독 성능점수가 아니라 조합의 상한과 확장·운영 여유에 반영한다.
- 케이스는 `maxGpuLengthMm`, `maxCpuCoolerHeightMm`, `maxPsuLengthMm`와 Tool의 실제 부품 치수 비교를 우선 사용한다. 여유가 충분하면 점수 차이는 작고, 한계에 가까워질수록 cap이 강하게 적용된다. airflow는 명시 필드가 있을 때만 보조 근거다.
- 점수 조언에서 이어지는 케이스 추천은 현재 제품을 제외하는 것만으로 충분하지 않다. 현재 견적을 후보별로 재평가해 케이스 cap과 최종 종합 점수가 모두 상승하고 blocking Tool `FAIL`이 없는 후보만 개선안으로 제시한다.
- RAM/STORAGE/PSU/CASE/COOLER/MOTHERBOARD는 저장된 `parts.attributes`와 Tool 결과를 사용한다.
- FPS seed는 근거 신뢰도에만 보탬이 되며 정확 FPS 보장으로 쓰지 않는다.
- 누락 데이터는 값을 지어내지 않고 `근거 신뢰도`와 cap/경고로 반영한다.

## 화면 표시 권장

- 1차 표시: `score / 1000`, `grade`, `label`
- AI 설명 카드: 총점, 강점 최대 3개, 주의점 최대 3개, 개선 우선순위 최대 3개. 내부 component 배점이나 raw normalized score는 사용자에게 표시하지 않는다.
- 조언 우선순위: 호환/전력/장착 FAIL → 점수 cap → CPU/GPU 불균형 → Tool WARN → 누락 부품/근거 부족.
- 배치도·실장도·3D의 문제 버튼은 category/tool을 focus 힌트로 보낼 수 있으나, 서버가 현재 Tool 결과에 존재하는 focus만 받아들인다.
- 보조 표시: 차단 사유, 요청 예산 적합도
- 곡선 그래프 marker:
  - 400: 검토 필요
  - 600: 기본형
  - 750: 균형형
  - 850: 고성능
  - 930: 끝판왕

## 주의

- 이 점수는 사용자 이해를 돕기 위한 종합 지표다.
- 실제 주문/결제/배송/AS 가능 여부를 대체하지 않는다.
- 부품 가격, seed, Tool 규칙이 갱신되면 같은 조합의 점수도 달라질 수 있다.
- 점수 설명의 사실과 개선 우선순위는 Tool이 확정한다. `gpt-5.4-mini`는 새 수치·제품·원인을 만들지 않고 표현만 다듬으며, 실패하면 Tool 기반 기본 문장을 그대로 사용한다.
