# BuildGraph AWS 부하 테스트 계획

## 1. 문서 목적

이 문서는 BuildGraph Green AWS 환경에서 과부하 테스트를 안전하고 재현 가능하게 수행하기 위한 실행 기준이다.

다음 자료를 함께 반영했다.

- `docs/aws-infrastructure*.md` 전체 문서
- `docs/AWS_LOAD_TEST_PREFLIGHT.md`
- `docs/SERVER_LOAD_TEST_GUIDE.md`
- `infra/k6/server-workload.js`, `infra/k6/smoke.js`
- 첨부된 부하 테스트 피드백

핵심 목표는 단순히 많은 요청을 보내는 것이 아니다. 아래 질문에 수치로 답할 수 있어야 한다.

1. 어느 동시 사용자 수와 RPS부터 지연이 증가하는가?
2. 최초 병목은 API, JVM, DB, Redis, RabbitMQ, XGBoost 중 어디인가?
3. 장애가 발생하지 않더라도 성능 저하 후 정상 상태로 회복하는가?
4. 병목 수정 전후에 같은 조건으로 얼마나 개선됐는가?
5. 1시간 이상 실행해도 메모리 누수, 연결 고갈, 큐 적체가 없는가?

> 프로젝트 목표는 비LLM API p95 500ms 이하, 에러율 1% 이하, 2주차 동시 300명, 4주차 동시 1,000명 검증이다. 단, `1,000 VU`와 `1,000 RPS`는 서로 다른 의미이므로 별도로 측정한다.

## 2. 결론

현재 `infra/k6/server-workload.js`를 Green 환경에 그대로 실행하는 것만으로는 최종 과부하 검증이 되지 않는다.

기존 스크립트는 빠른 회귀 검사에는 유용하지만 다음 한계가 있다.

- 모든 VU가 동일한 사용자 계정을 사용한다.
- 각 반복에서 API 하나를 독립적으로 무작위 호출하므로 실제 A–Z 사용자 흐름이 아니다.
- 정상 요청, 의도된 실패 요청, 계약 오류가 같은 실패율에 섞인다.
- 현재 `ai_fast` 요청은 실제 LLM이나 2초 Mock Worker가 아니라 서버 내부 fast-path다.
- 부품 목록이 일부 카테고리와 첫 페이지 중심이라 Redis/DB 캐시가 과도하게 유리할 수 있다.
- 원격 AWS 실행 시 PowerShell 러너가 Green EC2의 JVM/호스트 자원을 수집하지 못한다.
- 기존 Load/Stress/Spike는 VU 기반이고 Capacity만 도착률 기반이어서 사용자 수와 RPS 용량을 함께 설명할 수 없다.

따라서 아래 순서로 진행한다.

```text
배포 무결성 확인
→ 관측 환경 및 테스트 데이터 준비
→ Smoke
→ RPS 기준선/용량 측정
→ 상태 기반 A–Z 사용자 흐름 측정
→ 1시간 Soak
→ Spike 및 조건부 200/500/1,000 VU
→ 병목 수정
→ 동일 조건 재검증
```

## 3. 현재 AWS 구조와 테스트 해석

2026-07-14 감사 문서 기준 Green 환경은 다음과 같다.

```text
사용자/k6
  → Green CloudFront
    → /api/*, /ws/*
      → Green EC2 Nginx
        → Spring Boot API
        → XGBoost 컨테이너
          ├─ RDS PostgreSQL 16.14, db.t4g.small, Single-AZ
          ├─ ElastiCache Redis 7.1, cache.t4g.small, 단일 노드
          └─ Amazon MQ RabbitMQ, mq.m7g.medium, single-instance
```

주요 조건:

- Green CloudFront: `d2qhd7deuwmlln.cloudfront.net`
- Green EC2: `t3.medium` 단일 인스턴스
- ALB와 Auto Scaling Group이 없다.
- CloudFront가 EC2 Nginx를 직접 origin으로 사용한다.
- RDS, Redis, RabbitMQ도 각각 단일 인스턴스/노드 구성이다.
- WAF 규칙은 모두 `Count` 상태이므로 비정상 요청도 origin까지 도달할 수 있다.
- RDS Performance Insights는 활성화돼 있지만 Enhanced Monitoring은 비활성화돼 있다.
- CloudWatch 로그 그룹은 있으나 주요 장애 알람은 준비되지 않았다.

따라서 “애플리케이션은 수평 확장되므로 DB만 보면 된다”는 가정은 현재 구조에 맞지 않는다. 단일 EC2의 CPU credit, JVM heap, Tomcat/Hikari 연결, Nginx, Docker 자원도 DB와 동시에 관찰해야 한다.

## 4. 실행 전 필수 통과 조건

아래 항목이 하나라도 충족되지 않으면 본 부하 테스트를 시작하지 않는다.

### 4.1 배포 무결성

- [ ] ECR 이미지 불일치 방지 hotfix가 실제 Green 배포 브랜치에 반영됐다.
- [ ] 배포 manifest의 API/XGB 이미지 URI와 실행 중인 컨테이너의 image digest가 일치한다.
- [ ] `/api/health`가 DB 연결을 포함해 정상이다.
- [ ] API 배포 workflow의 OIDC, ECR, SSM 경로를 검증했다.
- [ ] Green 롤백 훈련을 1회 완료했다.
- [ ] 테스트 중 배포, 이미지 교체, 환경변수 변경을 금지했다.

이미지가 다르면 성능 비교 자체가 무효다. 각 실행 manifest에 Git SHA, 요청한 image URI, 실제 실행 image digest를 모두 남긴다.

### 4.2 데이터와 복구

- [ ] RDS 수동 snapshot을 만들고 복구 가능 여부를 확인했다.
- [ ] 테스트 전용 사용자 계정 풀을 만들었다.
- [ ] 테스트 데이터에 고유 prefix 또는 run ID를 붙였다.
- [ ] 테스트가 만든 견적, 알림, 문의, 세션을 run ID 기준으로 삭제할 수 있다.
- [ ] 운영 사용자와 운영 알림 발송에 영향이 없도록 격리했다.
- [ ] 가격 수집, 메일 발송, 정기 삭제 작업 등 scheduled job의 실행 여부를 기록했다.

현재 문서상 RDS deletion protection은 꺼져 있고 Redis snapshot retention은 0이다. 테스트 전에 복구 지점을 별도로 확보해야 한다.

### 4.3 관측 가능성

- [ ] k6 원시 결과와 summary JSON을 저장한다.
- [ ] EC2 CPU, CPU credit, 메모리, swap, 디스크, 네트워크를 수집한다.
- [ ] 컨테이너 CPU/메모리, 재시작 횟수, OOM 이벤트를 수집한다.
- [ ] JVM heap, GC pause, thread, Tomcat thread, Hikari active/pending/timeout을 수집한다.
- [ ] RDS CPU, CPU credit, 연결 수, read/write latency, IOPS, locks, Top SQL, wait event를 수집한다.
- [ ] Redis CPU, 메모리, 연결, hit rate, eviction을 수집한다.
- [ ] RabbitMQ queue depth, publish/ack rate, consumer 수, unacked/retry를 수집한다.
- [ ] XGBoost 응답시간, 오류율, 컨테이너 CPU/메모리를 수집한다.
- [ ] Nginx access log에 request time과 upstream response time을 남긴다.

현재 `application.yml`은 Actuator `health,info`만 노출하고 Dockerfile은 별도 JVM 옵션 없이 `java -jar`로 실행한다. Prometheus/JMX 또는 동등한 비공개 수집 경로를 먼저 준비해야 JVM과 connection pool 병목을 판정할 수 있다. 메트릭 endpoint는 CloudFront에 공개하지 않는다.

### 4.4 안전 조건

- [ ] 부하 발생기는 Green API EC2와 분리한다.
- [ ] 테스트 담당자, 시작/종료 시각, 중단 권한자를 정한다.
- [ ] CloudWatch dashboard와 로그를 동시에 볼 담당자를 정한다.
- [ ] 비용 한도와 실제 LLM 최대 호출 수를 정한다.
- [ ] 테스트 창 동안 실제 사용자가 받는 영향과 롤백 절차를 공유한다.

부하 발생기를 Green API EC2에서 실행하면 발생기 CPU와 네트워크가 서버 자원을 오염시키므로 금지한다. RDS도 외부 공개로 바꾸지 않는다. DB 직접 테스트가 필요하면 VPC 내부의 별도 임시 인젝터에서 수행한다.

## 5. 부하 모델 정의

### 5.1 VU와 RPS를 분리한다

- **VU**: 동시에 흐름을 수행하는 가상 사용자 수
- **RPS**: 서버에 실제로 도착하는 초당 HTTP 요청 수
- **iteration/s**: 전체 사용자 흐름 또는 스크립트 반복 완료 수

대략적인 관계는 다음과 같다.

```text
예상 RPS ≈ VU 수 ÷ (평균 응답시간 + 사용자 생각시간)
```

같은 100 VU라도 생각시간이 0.2초인 테스트와 3초인 테스트는 서버 부하가 크게 다르다. 따라서 모든 보고서에 VU, 실제 RPS, think time을 함께 기록한다.

### 5.2 요청 구성

첨부 피드백에 따라 시나리오 테스트와 무작위 테스트를 완전히 분리하지 않고, 한 사용자 흐름 안에서 다음 비율로 섞는다.

| 구분 | 비율 | 목적 |
| --- | ---: | --- |
| 정상 A–Z 흐름 | 85% | 실제 사용자 여정과 상태 전이 검증 |
| 유효한 무작위 행동 | 10% | 카테고리, 페이지, 정렬, 부품, 읽기 순서를 다양화 |
| 의도된 실패 요청 | 5% | 400/401/403/404/409 처리와 회복 검증 |

무작위 요청도 현재 사용자 상태에서 가능한 동작 중에서 선택한다. 상태를 무시한 완전 랜덤 호출은 의도된 실패 요청에서만 사용한다.

### 5.3 테스트 사용자

- VU마다 서로 다른 계정을 사용한다.
- 계정은 테스트 전에 생성하고 계정 ID와 VU를 고정 매핑한다.
- access/refresh token도 VU별로 독립 관리한다.
- 15분 access token TTL을 넘는 테스트에서는 refresh 흐름을 반드시 포함한다.
- 동일 계정을 모든 VU가 공유하는 방식은 사용자 격리, row lock, 세션 동작을 왜곡하므로 금지한다.

## 6. 정상 A–Z 사용자 흐름

실제 API 계약에 맞춰 아래 흐름을 상태 머신으로 구현한다. 각 단계의 응답에서 받은 public ID를 다음 단계에 사용하며 하드코딩된 공용 ID를 공유하지 않는다.

1. 로그인
2. `/api/auth/me` 사용자 확인
3. 부품 카테고리, 페이지, 정렬 조건을 무작위로 조회
4. 견적 draft 생성
5. CPU, GPU, 메모리, 메인보드, 저장장치, PSU 등을 순차 추가
6. 일부 사용자는 부품 교체 또는 제거 후 다시 추가
7. compatibility, power, size, performance, price Tool 검증
8. 요구사항 생성 및 추천 요청
9. 추천 결과와 경고 조회
10. 견적 저장
11. 채팅 결과에서 build 생성
12. 견적 이력 조회
13. 저장된 견적의 부품 한 개 변경 후 영향 비교
14. 일부 사용자는 가격 알림 등록/조회
15. 일부 사용자는 조립 요청 또는 지원/AS 흐름 조회
16. WebSocket 대상 사용자는 연결 유지, 메시지 수신, 정상 종료

모든 VU가 매번 전체 쓰기 흐름을 실행하면 실제 사용 패턴보다 쓰기 비중이 과도해진다. 정상 흐름 내부 비율은 실제 화면 사용에 맞춰 읽기 중심으로 두되, 최소한 draft 생성부터 저장까지의 완전한 흐름이 지속적으로 포함돼야 한다.

권장 think time:

- 화면 조회 사이: 1~3초
- 부품 선택/변경: 2~5초
- 추천 결과 확인: 3~8초
- 시스템 용량 한계를 찾는 RPS 테스트: think time 대신 arrival-rate로 통제

## 7. 무작위 및 실패 요청

### 7.1 유효한 무작위 행동

- 부품 카테고리를 전체 지원 값에서 무작위 선택
- 페이지 번호와 정렬 조건 변경
- 캐시된 첫 페이지뿐 아니라 여러 페이지 조회
- draft에 넣는 부품 순서 변경
- 일부 VU는 교체, 일부는 제거, 일부는 그대로 저장
- 조회/쓰기 비율과 endpoint별 가중치를 명시
- 캐시 warm/cold 실행을 분리해 기록

### 7.2 의도된 실패 요청

다음은 예상 상태 코드를 별도 metric으로 기록한다.

| 사례 | 예상 결과 예시 |
| --- | --- |
| 토큰 없이 보호 API 호출 | 401 |
| 만료/변조 토큰 | 401 |
| 존재하지 않는 public ID | 404 |
| 다른 사용자의 리소스 접근 | 계약에 따른 403 또는 404 |
| 필수 필드 누락/잘못된 enum | 400 |
| 호환되지 않는 상태 전이/중복 요청 | 계약에 따른 400 또는 409 |
| 과도한 page/size 또는 비정상 문자열 | 계약에 따른 400 또는 안전한 빈 결과 |

의도한 4xx는 서버 실패율에 포함하지 않는다. 다음 세 지표를 분리한다.

- `expected_rejections`: 예상한 4xx가 정확히 반환된 비율
- `unexpected_business_errors`: 정상 요청에서 발생한 4xx/5xx
- `contract_errors`: 상태 코드 또는 응답 shape가 계약과 다른 비율

현재 `server-workload.js`의 `verify()`는 기본적으로 HTTP 200만 성공으로 보기 때문에 이 분리가 가능하도록 확장해야 한다.

## 8. 단계별 실행 계획

각 단계는 이전 단계가 합격한 경우에만 진행한다.

### 8.1 Stage 0 — 배포 및 계측 확인

기간: 약 10분

- 실제 실행 image digest와 배포 manifest 비교
- health, 로그인, refresh, parts, draft, Tool, 추천, 저장을 각 1회 호출
- CloudWatch, 로그, JVM, RDS, Redis, MQ metric 유입 확인
- 테스트 계정과 cleanup 동작 확인

### 8.2 Stage 1 — Smoke

기간: 2분, 5 VU

- 전체 A–Z 흐름을 낮은 부하로 실행
- 계약 오류, 인증 갱신, 데이터 cleanup 검증
- 실패가 하나라도 있으면 원인 해결 후 다시 시작

기존 `infra/k6/smoke.js`는 30초 동안 일부 endpoint만 호출하므로 최종 Smoke에는 부족하다. 새 상태 기반 workload의 `smoke` profile을 사용한다.

### 8.3 Stage 2 — RPS 기준선 및 용량

총 60분, arrival-rate 기반:

| 구간 | 목표 RPS | 시간 |
| --- | ---: | ---: |
| 기준선 | 2 | 5분 |
| Ramp 1 | 5 | 10분 |
| Ramp 2 | 10 | 10분 |
| Ramp 3 | 20 | 10분 |
| Ramp 4 | 40 | 10분 |
| Ramp 5 | 80 | 10분 |
| 회복 확인 | 5 | 5분 |

각 구간에서 p50/p95/p99, 실제 RPS, dropped iteration, 오류율과 모든 서버 metric을 같은 시간축으로 비교한다.

80 RPS까지 안정적이면 100, 150, 200 RPS를 별도 10분 구간으로 추가할 수 있다. 이미 중단 기준을 넘었다면 더 높은 부하를 보내지 않는다.

### 8.4 Stage 3 — 상태 기반 동시 사용자

정상 A–Z + 유효 랜덤 + 의도된 실패를 혼합한다.

| 단계 | VU | 유지 시간 |
| --- | ---: | ---: |
| 1 | 10 | 10분 |
| 2 | 30 | 10분 |
| 3 | 50 | 10분 |
| 4 | 80 | 10분 |
| 5 | 100 | 10분 |

피드백에서 언급된 “약 30명부터 지연”, “약 80명에서 서버 종료”를 우선 재현하되 과거 수치를 사실로 가정하지 않고 새 결과로 검증한다.

100 VU까지 합격한 경우에만 아래 확장 단계를 수행한다.

| 확장 단계 | VU | 유지 시간 | 조건 |
| --- | ---: | ---: | --- |
| 1 | 200 | 10분 | 100 VU 합격 |
| 2 | 300 | 10분 | 200 VU 합격 |
| 3 | 500 | 10분 | 300 VU 합격 |
| 4 | 1,000 | 10분 | 500 VU 합격, 발생기 용량 충분 |

1,000 VU 목표는 앞 단계가 안정적인 경우의 검증 목표이지 무조건 한 번에 가해야 할 시작 부하가 아니다. k6 부하 발생기의 CPU/네트워크가 먼저 포화되지 않았는지도 함께 확인한다.

### 8.5 Stage 4 — Soak

기간: 최소 1시간

- 마지막 안정 처리량의 60~70% 부하를 유지
- 5분 단위로 p95/p99, 오류율, heap, thread, DB connection, queue depth를 집계
- access token 갱신을 실제로 통과
- 메모리, thread, connection, queue가 시간에 따라 계속 증가하는지 확인
- 종료 후 5분 안에 기준선으로 회복하는지 확인

5분 Soak는 설정 확인일 뿐 내구성 근거로 사용하지 않는다. 운영 인증이 필요하면 동일 조건으로 2시간 이상 재실행한다.

### 8.6 Stage 5 — Spike

예시:

```text
10 VU 2분
→ 100 VU 10초 이내 상승
→ 100 VU 3분 유지
→ 10 VU 5분 회복 관찰
```

목표는 순간 최대 처리량이 아니라 timeout, restart, queue 폭증 없이 회복하는지 확인하는 것이다.

## 9. LLM과 RabbitMQ 테스트

LLM 테스트는 서버/DB 기본 용량 테스트와 분리해서 결과를 해석한다.

### 9.1 Mode A — 비LLM 기준선

- 실제 OpenAI 호출을 제외한다.
- compatibility, power, size, performance, price 등 결정적 Tool과 일반 API를 측정한다.
- API/JVM/DB/Redis/XGBoost의 기본 병목을 찾는다.

현재 `server-workload.js`의 `ai_fast`는 `BOARD_PART_FOCUS` fast-path를 사용해 실제 LLM과 RabbitMQ 대기시간을 측정하지 않는다.

### 9.2 Mode B — 2초 LLM Mock

테스트 전용 OpenAI 호환 stub을 준비한다.

- `OPENAI_BASE_URL`을 테스트 stub으로 지정
- 요청마다 약 2초 뒤 유효하고 고정된 구조의 응답 반환
- timeout, 429, 500 응답을 작은 비율로 주입 가능한 별도 profile 제공
- 운영 비밀키나 실제 OpenAI 비용 없이 서버의 timeout/queue/동시성 검증

Mock 결과는 다음처럼 분해한다.

```text
전체 응답시간
= 큐 대기시간
+ worker 처리시간(약 2초)
+ API/DB 후처리시간
```

LLM Mock Queue의 목표는 p95 대기 30초 이하이다. 비LLM p95 500ms 기준과 같은 표에서 합격/실패를 판단하지 않는다.

### 9.3 Mode C — 실제 LLM

- 동시성 1 → 2 → 4 → 8 → 16 순서로 제한
- 전체 실제 호출 수는 100~300건 이내
- token, 비용, 429, timeout, 응답시간, 품질 오류를 기록
- 1만 요청 환산 비용이 100달러 이하인지 계산

실제 LLM 테스트는 서버 최대 용량 시험이 아니라 비용과 외부 의존성 지연 확인용이다.

### 9.4 RabbitMQ 단독 확인

- publish rate, consume rate, queue depth, unacked, retry/dead-letter를 기록
- worker 수보다 빠른 요청이 들어왔을 때 queue 증가 기울기를 측정
- 입력 중단 후 queue가 정상적으로 0까지 감소하는지 확인
- consumer 재시작 후 메시지 유실/중복 처리 여부를 확인

## 10. DB 집중 테스트

전체 사용자 흐름에서 병목 후보를 먼저 찾고 RDS Performance Insights의 Top SQL과 wait event를 기준으로 DB 직접 테스트 대상을 고른다.

우선 점검 항목:

- 부품 카테고리/정렬/페이지 조회
- 추천과 Tool 검증에서 반복되는 조회
- 견적 저장/이력 조회
- 가격 알림 조회
- N+1 쿼리
- index 미사용 또는 잘못된 복합 index 순서
- lock wait, long transaction
- Hikari connection pending/timeout

DB 직접 부하는 다음 조건을 지킨다.

- 운영 RDS를 public으로 전환하지 않는다.
- Green API EC2에서 `pgbench`를 실행하지 않는다.
- VPC 내부 별도 인젝터를 사용한다.
- 가능하면 snapshot에서 복제한 테스트 DB에 custom query pack을 실행한다.
- `pgbench` 결과와 애플리케이션 E2E 결과를 같은 의미로 해석하지 않는다.

DB가 병목이면 Top SQL 실행계획, 호출 횟수, 평균/총 소요시간을 보존한 뒤 index/query 수정 전후를 같은 데이터셋으로 비교한다.

## 12. 합격 및 중단 기준

### 12.1 기본 합격 기준

- 비LLM 정상 API p95: 500ms 이하
- 정상 요청 오류율: 1% 이하
- 안정 구간 권장 오류율: 0.1% 이하
- 계약 오류율: 0%
- dropped iteration: 0
- OOM, 컨테이너 재시작, DB connection timeout: 0
- 의도된 실패 요청의 예상 status 일치율: 99% 이상
- 부하 종료 후 5분 이내 latency와 queue가 기준선으로 회복
- Soak 중 heap/thread/connection/queue에 지속적인 우상향이 없음

endpoint 특성상 500ms가 불가능한 작업은 사전에 별도 SLO를 정하고 보고서에서 분리한다. 2초 LLM Mock은 `약 2초 지연 + 서버 오버헤드`로 판단한다.

### 12.2 즉시 중단 기준

다음 중 하나라도 발생하면 부하를 더 높이지 않고 중단한다.

- p95가 2초를 초과한 상태로 3분 지속
- 정상 요청 오류율이 1%를 초과한 상태로 2분 지속
- health check가 연속 2회 실패
- API/XGBoost 컨테이너 재시작 또는 OOM
- EC2/RDS 메모리 85% 이상 또는 swap 사용 시작
- 디스크 사용률 80% 이상
- CPU 85% 이상이 5분 지속되면서 latency가 함께 증가
- DB connection pool 80% 이상이 지속되거나 acquisition timeout 발생
- RabbitMQ queue가 계속 증가하고 입력 중단 후에도 감소하지 않음
- 데이터 오염, 실제 알림 발송, 비용 급증 등 운영 영향 발생

중단 직전 10분과 중단 후 회복 5분의 metric과 로그를 반드시 보존한다.

## 13. 실행 결과 보존 형식

실행마다 고유 run ID를 만든다.

```text
YYYYMMDD-HHmm-{gitShortSha}-{profile}
```

권장 구조:

```text
infra/k6/reports/runs/{runId}/
  manifest.json
  k6-summary.json
  k6-console.log
  endpoint-metrics.json
  aws-metrics/
  container-stats.csv
  jvm-metrics.csv
  nginx-access.log
  application.log
  rds-top-sql.md
  events.md
  result.md
```

`manifest.json` 필수 항목:

- 시작/종료 시각과 타임존
- 대상 CloudFront/origin
- Git SHA
- 배포 manifest image URI
- 실제 실행 image digest
- profile, VU, 목표/실제 RPS, think time
- 정상/랜덤/실패 요청 비율
- cache warm/cold 상태
- 테스트 계정/데이터 버전
- API/JVM/DB/Redis/MQ/XGB 설정
- scheduled job 활성 여부
- 실행자와 중단 사유

병목 수정 전후는 같은 데이터, 계정, profile, 시간, cache 조건으로 실행한다.

보고서 결론은 다음 형식으로 작성한다.

```text
관찰된 병목
→ 근거 지표와 Top SQL/로그
→ 적용한 변경
→ 동일 조건 재실행 결과
→ p95/p99, RPS, 오류율, 자원 사용 개선폭
→ 남은 위험
```

## 14. 기존 도구 보완 항목

### P0 — 본 테스트 전에 필요

- [ ] VU별 사용자 계정 풀 지원
- [ ] 상태 기반 A–Z scenario 구현
- [ ] normal/random/negative 비율과 별도 metric 구현
- [ ] endpoint별 예상 status 지원
- [ ] category/page/sort/part 데이터 다양화
- [ ] macOS/Linux에서 실행 가능한 AWS suite runner 추가
- [ ] 원격 EC2/JVM/RDS/Redis/MQ metric 수집 추가
- [ ] run manifest에 실제 image digest 저장
- [ ] 요약기에 새 profile과 expected rejection 지표 추가

### P1 — LLM/내구 검증 전에 필요

- [ ] 2초 OpenAI 호환 Mock 서버 또는 Worker
- [ ] queue wait/worker/total latency 분리
- [ ] WebSocket 장기 연결 profile
- [ ] 1시간 이상 Soak 구간별 추세 보고
- [ ] 실제 LLM 호출 수와 비용 상한

기존 PowerShell runner는 로컬 listening process의 자원 수집을 전제로 하므로 macOS에서 원격 Green URL만 지정하면 AWS JVM/호스트 metric을 얻지 못한다. k6 결과만으로 서버 병목을 판단하지 않는다.

## 15. 우선 실행 순서

시간이 제한적이면 다음 순서를 지킨다.

1. ECR image/manifest 불일치 hotfix 배포 및 digest 검증
2. snapshot, 테스트 계정, cleanup, 중단 기준 준비
3. EC2/JVM/Hikari/RDS/Redis/MQ 관측 준비
4. 5 VU Smoke
5. 2→80 RPS 1시간 기준선
6. 10→100 VU 혼합 A–Z 테스트
7. 가장 큰 병목 한 개 수정
8. 동일 조건 재실행 및 전후 비교
9. 마지막 안정 부하의 60~70%로 1시간 Soak
10. 시간이 남고 앞 단계가 합격한 경우 200→300→500→1,000 VU

300명/1,000명 숫자만 달성하고 원인을 설명하지 못하는 것보다, 최초 병목과 수정 효과를 같은 조건으로 증명하는 것이 우선이다.
