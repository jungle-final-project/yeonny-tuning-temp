#### admin demo 계정 흐름

- 백엔드 seed 기준으로 이메일이 `admin`으로 시작하면 관리자 계정으로 취급한다.
- `admin@example.com`으로 로그인하면 `UserSeed.login()`이 `role: "ADMIN"`과 `token: "demo-jwt-admin"`을 반환한다.
- `user@example.com`으로 로그인하면 `role: "USER"`와 `token: "demo-jwt-user"`를 반환한다.
- 프론트 공통 API helper는 로그인 성공 token을 `buildgraph.token` key로 localStorage에 저장한다.
- `RequireAdmin`은 먼저 `buildgraph.token`이 있는지 확인한다.
- token이 없으면 `/api/auth/me`를 호출하지 않고 "관리자 권한이 필요합니다" 안내 화면을 보여준다.
- token이 있으면 `GET /api/auth/me`를 호출해 현재 사용자 role을 확인한다.
- `role: "ADMIN"`이면 `/admin/*` 화면 접근을 허용한다.
- `role: "USER"`이거나 `auth/me` 확인에 실패하면 관리자 화면 대신 "관리자 권한이 필요합니다" 안내 화면을 보여준다.

#### 담당 경계

- 5번은 위 demo token 정책과 `RequireAdmin` guard 동작을 문서화한다.
- 로그인 화면에서 `admin@example.com`을 직접 입력해 로그인할 수 있게 form state를 연결하는 작업은 1번 담당이다.
- admin demo 계정 안내는 화면에 추가하지 않고 문서에만 둔다.

### 2.1 현재 skeleton 기준 확인

- [x] 현재 구현이 Sprint 1 skeleton이라는 점을 문서화한다.

### 2.2 role 기반 권한 판단 정책

- [x] 백엔드 `AuthResponse.user.role` 값을 기준으로 권한 판단할지 결정한다.
- [x] `GET /api/auth/me` 응답의 `role`을 기준으로 권한 판단할지 결정한다.
- [x] localStorage token 문자열 검사 방식을 언제 제거할지 정한다.
- [x] 관리자 권한 확인 실패 시 화면 표시 방식과 redirect 방식을 비교한다.
- [x] Sprint 1에서는 안내 화면 유지 여부를 결정한다.

#### 결정 메모

- 관리자 route guard는 `AuthResponse.user.role` 대신 `GET /api/auth/me` 응답의 `role`을 기준으로 판단한다.
- localStorage token 문자열에 `demo-jwt-admin`이 포함되어 있는지 직접 검사하는 방식은 제거한다.
- localStorage는 로그인 token 존재 여부를 빠르게 확인하는 용도로만 사용한다.
- 권한 확인 실패 시 redirect하지 않고 "관리자 권한이 필요합니다" 안내 화면을 유지한다.

### 2.3 401/403 정책 확정

- [x] 로그인하지 않은 사용자는 401로 볼지 정한다.
- [x] 로그인했지만 관리자 권한이 없으면 403으로 볼지 정한다.
- [x] 프론트에서 401과 403을 다른 메시지로 보여줄지 정한다.
- [x] API error response 공통 schema가 필요한지 확인한다.
- [x] 정책이 확정되면 `docs/openapi.yaml` 반영 필요 여부를 확인한다.

#### 결정 메모

- 로그인하지 않은 사용자는 `401 UNAUTHORIZED`로 본다.
- 로그인했지만 관리자 권한이 없는 사용자는 `403 FORBIDDEN`으로 본다.
- 프론트는 401일 때 로그인 필요 메시지, 403일 때 관리자 권한 없음 메시지를 보여준다.
- API error response는 `{ code, message }` 공통 schema를 사용한다.
- 이 정책은 `docs/openapi.yaml`의 `ErrorResponse`, `UnauthorizedError`, `ForbiddenError`, `/api/auth/me`, `/api/admin/*` 응답 계약에 반영한다.

### 2.4 OpenAPI 반영 여부 확인

- [x] `/api/admin/*` 경로에 bearerAuth 표시가 필요한지 확인한다.
- [x] `/api/auth/me` 응답 schema 상세화가 필요한지 확인한다.
- [x] 401/403 response를 OpenAPI에 추가할지 결정한다.
- [x] 공통 error schema를 추가할지 결정한다.
- [x] 결정 전에는 임의로 OpenAPI를 확장하지 않는다.

#### 결정 메모

- `/api/admin/*` 경로에는 `bearerAuth` 보안 요구사항을 표시한다.
- `/api/auth/me`의 200 응답은 `User`, 401 응답은 `ErrorResponse`로 명시한다.
- `/api/admin/*`의 401/403 응답은 공통 response component로 명시한다.
- 정책이 사용자에 의해 확정되었으므로 OpenAPI 확장을 진행한다.
- OpenAPI에는 `ErrorResponse`, `UnauthorizedError`, `ForbiddenError`, `/api/auth/me`, `/api/admin/*` 보안 요구사항과 401/403 응답을 반영했다.
