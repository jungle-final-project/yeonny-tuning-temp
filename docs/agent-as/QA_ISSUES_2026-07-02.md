# Agent AS QA 발견 문제 기록

작성일: 2026-07-02
기준 커밋: `50065bac5b26f62aaf68986dbe9039171fe3c547`

이 문서는 Agent AS runtime QA와 exe download 브랜치 병합 QA 중 발견한 문제를 별도로 남긴 기록이다. 해결된 항목은 수정 커밋 기준으로 표시하고, 중대한 환경/운영 이슈는 임의로 고치지 않고 남긴다.

## 해결 완료

| 항목 | 증상 | 원인 | 처리 |
|---|---|---|---|
| multipart upload body caching | `POST /api/agent/log-uploads`에서 multipart body가 필터 단계에서 소모되어 controller upload가 실패할 수 있었다. | Agent idempotency filter가 multipart 요청 body를 일반 요청처럼 캐싱했다. | multipart 요청은 service 레벨 idempotency로 처리하도록 수정했다. |
| upload idempotency replay/conflict | 같은 upload key 재시도와 다른 body 충돌 처리가 안정적으로 분리되지 않았다. | upload 처리와 idempotency 저장/replay 책임이 분산되어 있었다. | upload service에서 replay/conflict 기준을 명확히 처리했다. |
| SQL timestamp parameter | AS/Agent runtime 흐름 중 timestamp parameter 타입 문제가 발생했다. | JDBC parameter 타입이 DB timestamp 기대와 맞지 않았다. | timestamp parameter 전달 방식을 수정했다. |
| gzip upload 재시도 hash 불안정 | 같은 JSONL 내용으로 gzip을 다시 만들면 idempotency hash가 달라질 수 있었다. | gzip mtime 기본값이 매번 달라졌다. | Agent gzip 생성 시 deterministic mtime을 사용하도록 수정했다. |
| missing header 오류 응답 | `Authorization`, `Idempotency-Key` 누락 시 공통 오류 응답 형식이 일관되지 않았다. | security/filter 예외 처리 경로가 공통 `ApiErrorResponse` 기준과 달랐다. | 누락 header validation과 공통 오류 응답을 맞췄다. |
| 사용자/관리자 ticket decision 표시 | backend 응답에는 `analysisStatus`, `reviewStatus`, `supportDecision`이 있으나 화면에 보이지 않았다. | 사용자/관리자 AS 상세 UI가 새 decision 필드를 표시하지 않았다. | `/support/{ticketId}`, `/admin/as-tickets/{ticketId}`에 핵심 decision 필드 표시를 추가했다. |
| 모바일 AS ticket 정보 가독성 | Agent AS 상세 화면에서 표/텍스트가 모바일 폭에서 읽기 어려웠다. | DataTable과 grid가 긴 값을 충분히 줄바꿈하지 못했다. | Agent AS 본문 중심으로 줄바꿈과 grid 표시를 보정했다. |
| exe download 진입점 부재 | 웹 fallback AS 접수 화면에서 PC Agent exe를 받을 수 없었다. | feature branch 전까지 공개 다운로드 링크가 없었다. | `/support/new`에 `PC Agent 다운로드`, 실행 안내 링크를 추가했다. |
| PowerShell 생성 config BOM | QA 중 PowerShell `Set-Content -Encoding UTF8`로 만든 Agent config를 CLI가 `Unexpected UTF-8 BOM`으로 읽지 못했다. | `read_config_json()`이 `utf-8`만 사용해 BOM이 붙은 JSON을 거부했다. | Agent config reader를 `utf-8-sig`로 바꾸고 BOM config unit test를 추가했다. |

## 남은 문제와 주의점

| 항목 | 현재 상태 | 영향 | 조치 기준 |
|---|---|---|---|
| 기존 Compose DB volume checksum mismatch | 이전 QA에서 기존 Compose DB volume의 Flyway V53/V54 checksum mismatch가 확인됐다. | 기존 로컬 volume으로 바로 기동하면 migration 검증에 실패할 수 있다. | 중대한 환경 이슈라 임의 삭제/repair하지 않았다. 필요 시 팀 동의 후 DB volume 재생성 또는 Flyway repair 절차를 별도 수행한다. |
| 기존 QA DB 재사용 시 fresh demo register 409 | `pcagent-postgres-runtime-qa`에 이미 등록된 demo device가 있는 상태에서 새 fingerprint로 register하면 `409 Agent device is already registered.`가 발생했다. | 기존 QA DB 상태와 demo activation token 정책이 섞인 문제다. | DB를 삭제하지 않는 원칙을 지키기 위해 기존 QA device의 registration refresh 경로로 runtime QA를 계속 진행했다. 별도 신규 등록 QA는 깨끗한 DB에서 수행한다. |
| custom API port의 supportUrl 변환 | `apps/pc-agent`는 기본 `8080 -> 5173` 변환은 유지하지만, QA용 `18080 -> 15173` 같은 custom mapping은 자동 변환하지 못한다. | custom port QA에서는 Agent 출력 URL을 수동으로 웹 포트에 맞춰 열어야 한다. | 운영 설정에서 web base URL을 별도 config로 분리하는 보강이 필요하다. |
| Playwright sandbox 네트워크 제한 | sandbox 기본 실행에서는 Vite proxy가 `EACCES`를 출력하며 timeout될 수 있었다. | 테스트 assertion 실패가 아니라 실행 환경의 네트워크 제한으로 보인다. | 권한 상승 실행에서 `74 passed`로 재검증했다. CI/로컬에서는 정상 reporter 종료 여부를 계속 확인한다. |
| Docker config 접근 경고 | `docker compose config --quiet` 실행 시 `C:\Users\whqja\.docker\config.json` 접근 경고가 출력됐다. | compose config 검증은 exit code 0으로 성공하지만 경고가 남는다. | 사용자 Docker config 권한 문제로 분리한다. |
| 관리자 supportDecision UI | 관리자 결정 저장은 API와 테스트로 검증되지만, 운영자가 입력하는 완성형 form UX는 아직 제한적이다. | 운영 사용성은 데모 수준이다. | 원격/방문지원 운영 화면 고도화 작업에서 처리한다. |
| Windows Service/tray/installer | 현재 exe는 개발용 PyInstaller 단일 실행 파일이다. | 완성형 사용자 서비스 설치/업데이트/상주 실행은 아직 아니다. | Inno Setup, Service, Tray, auto-update는 후속 운영 패키징 단계로 남긴다. |
